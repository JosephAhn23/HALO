Why teams add a gate

Three ways an LLM system fails without one

None of these throw an exception. That's the problem.

SILENT REGRESSION

Faithfulness drops, nothing turns red

A retriever tweak or prompt edit quietly drops faithfulness or context recall. No test fails — a support ticket does, three weeks later.

COST DRIFT

The router sends everything to the expensive model

A routing bug or a bad default sends low‑complexity queries to a frontier model. Nobody sees it until the invoice lands.

NO SIGN‑OFF

Merges happen on vibes

There's no artifact that says this build was actually evaluated — just a reviewer's gut feeling and a demo that looked fine.

Mechanism

Four steps, one PR comment

Threshold doesn't replace your eval harness — it enforces the result of one.

01 / BASELINE

Store your last known‑good run

Faithfulness, answer relevancy, context precision/recall, and cost‑per‑query from your current main branch become the baseline Threshold compares against.

02 / EVAL

Every PR runs the eval set

Your held‑out question set runs through RAGAS‑style scoring and the cost‑aware router on every push — no separate step to remember to run.

03 / COMPARE

Deltas are checked against tolerance

Each metric is compared to baseline within a tolerance you set (default ±0.03 absolute) and against a hard cost ceiling per 1k queries.

04 / VERDICT

Pass unblocks, fail explains why

A failing gate comments the exact metric, delta, and threshold that tripped on the PR — merge stays blocked until it's addressed or explicitly overridden.

# .github/workflows/ci.yml — this is the real action, not a mockup: # github.com/JosephAhn23/HALO/tree/main/.github/actions/gate - uses: JosephAhn23/HALO/.github/actions/gate@main id: gate with: scores: reports/ragas_scores.json baseline: eval/ragas_baseline.json tolerance: '0.03' - if: steps.gate.outputs.passed != 'true' run: exit 1
